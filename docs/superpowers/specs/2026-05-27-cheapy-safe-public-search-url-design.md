# Cheapy Safe Public Search URL Design

Date: 2026-05-27

## Summary

Add a simple optional `public_search_url` field to each Contract V1 flight
offer. The field is a safe provider public search URL, not a guaranteed checkout
or exact booking deeplink.

The URL is generated by Cheapy from normalized request and offer data. Cheapy
does not expose provider/browser internal URLs, raw deeplinks, cookies, headers,
request bodies, tokens, challenge URLs, session IDs, or raw provider payloads.

## Approved Direction

Use the simpler MVP shape:

```json
{
  "public_search_url": "https://..."
}
```

When Cheapy cannot build a safe public URL, the field is `null`.

This intentionally avoids a broader `links` object until Cheapy needs multiple
link types. It also avoids the name `booking_url`, because these URLs reopen a
public provider search and may not point to the exact fare checkout.

## Goals

- Make agent output more useful with clickable fare text.
- Keep URL semantics honest: public search URL only.
- Support current live providers:
  - `google_fli`
  - `traveloka`
  - `skyscanner`
- Keep provider internals separate from persistence.
- Preserve MCP stdout protocol cleanliness.
- Keep default tests offline.
- Keep the change additive for Contract V1 consumers.

## Non-Goals

- No exact fare checkout deeplinks.
- No booking flow automation.
- No provider raw URL passthrough.
- No Skyscanner `/transport_deeplink/...` exposure.
- No Browserless reintroduction.
- No cookies, headers, request bodies, tokens, challenge URLs, session IDs, raw
  provider payloads, or browser session data in Contract V1 or storage.
- No SQLite URL columns in `offer_observations`.
- No live provider calls in normal tests.

## Contract V1 Change

Add one optional field to `FlightOfferV1`:

```python
public_search_url: str | None = None
```

Validation requirements:

- `None` is allowed.
- Non-null values must pass the shared public URL validator.
- The field is additive. JSON clients that ignore new fields keep working;
  clients generated from Cheapy's strict schema need to regenerate from the
  updated schema. Contract/schema tests must be updated because Cheapy's public
  schema changes.

## URL Generation

Add a small public URL module, for example `cheapy/public_links.py`, with one
entrypoint:

```python
def attach_public_search_urls(
    request: SearchRequestV1,
    response: SearchResponseV1,
) -> SearchResponseV1:
    ...
```

The search service or search orchestration calls this after provider results
are normalized and ranked, before the response is returned or persisted.

The module owns:

- provider dispatch by `offer.provider`
- conversion from `SearchRequestV1` plus `FlightOfferV1` into provider-local
  public search URL inputs
- shared validation and fail-closed behavior

If a provider builder raises or validation rejects the URL, Cheapy sets
`public_search_url=None` for that offer and does not add a warning. Missing
links are expected fallback behavior, not provider failure.

## Provider Strategies

### Traveloka

Use the existing public search URL builder in `cheapy/providers/traveloka/urls.py`.

Current builder shape:

```text
https://www.traveloka.com/en-en/flight/fulltwosearch
  ?ap=SGN.BKK
  &dt=10-7-2026
  &ps=1.0.0
  &sc=ECONOMY
  &funnelSource=SEO-Homepage-SearchForm
```

Round trips append the return date in `dt`, for example:

```text
dt=10-7-2026.17-7-2026
```

Traveloka internal API paths such as `/api/v2/flight/search/initial` and
`/api/v2/flight/search/poll` are never exposed.

### Google Fli

Generate a Google Flights public search URL from Contract V1 data. This is a
public Google Flights search URL, not an upstream `fli` internal URL and not an
exact checkout deeplink.

Use a conservative URL such as:

```text
https://www.google.com/travel/flights?q=Flights%20from%20SGN%20to%20BKK%20on%202026-07-10
```

For round trips, include both dates in the query text. Passenger counts and
economy cabin may be included in query text only when doing so remains simple
and deterministic. Avoid opaque Google `tfs` encodings unless a tested builder
is added later.

No URL is read from upstream `fli` result objects.

### Skyscanner

Generate a public Skyscanner search URL from Contract V1 data, using the same
public route shape already used as the search referer:

```text
https://www.skyscanner.com.sg/transport/flights/sgn/bkk/260710/
```

Round trip:

```text
https://www.skyscanner.com.sg/transport/flights/sgn/bkk/260710/260717/
```

Optional query parameters can mirror the existing safe referer:

```text
adultsv2=1&cabinclass=economy&childrenv2=&ref=home&rtn=0
```

Skyscanner `/transport_deeplink/...` URLs stay internal to the provider's
actionability check and are not exposed or persisted. Those deeplinks can carry
opaque tracking/session query fields such as request or session identifiers.

## URL Safety Rules

Every non-null URL must pass validation:

- Scheme must be `https`.
- Host must match provider allowlist exactly:
  - Google: `www.google.com`
  - Traveloka: `www.traveloka.com`
  - Skyscanner: `www.skyscanner.com.sg`
- URL must not include userinfo.
- URL must not include a non-default port.
- URL must not include a fragment unless a provider explicitly needs and
  validates it. The MVP uses no fragments.
- Path must be in a provider public search allowlist:
  - Google: `/travel/flights`
  - Traveloka: `/en-en/flight/fulltwosearch`
  - Skyscanner: `/transport/flights/...`
- Path must not contain internal/API/challenge segments such as:
  - `/api/`
  - `/g/radar/`
  - `/transport_deeplink/`
  - `/challenge`
  - `/captcha`
  - `/login`
  - `/session`
- Query keys must be generated by Cheapy, not copied from raw provider
  responses.
- Query keys or values must not contain sensitive markers:
  - `token`
  - `cookie`
  - `header`
  - `body`
  - `payload`
  - `session`
  - `request_id`
  - `fps_session_id`
  - `q_ids`
  - `auth`
  - `jwt`
  - `challenge`

Validation fails closed by returning `None`.

## Storage Impact

Persist `public_search_url` only inside sanitized `search_runs.response_json`,
because `response_json` represents the returned Contract V1 response.

Do not add URL columns to `offer_observations`. Historical comparison,
watchlist decisions, itinerary fingerprints, and best-price summaries do not
need URLs.

Storage sanitizer should revalidate offer `public_search_url` values before
writing `response_json`. Invalid values are changed to `null`. Warning and
error details keep the existing aggressive URL redaction behavior.

`cheapy history show` may display the stored `public_search_url` in the response
object. `cheapy history list` remains unchanged unless a later UI explicitly
needs links in summaries.

## Output Impact

MCP and CLI JSON include `public_search_url` on each offer:

```json
{
  "price_amount": 4920000,
  "currency": "VND",
  "provider": "traveloka",
  "public_search_url": "https://www.traveloka.com/en-en/flight/fulltwosearch?..."
}
```

Agent markdown can render:

```md
[4,920,000 VND on Traveloka](https://www.traveloka.com/en-en/flight/fulltwosearch?...)
```

Fallback when `public_search_url` is `null`:

```md
4,920,000 VND on Traveloka
```

The Cheapy agent skill can be updated to tell agents to link prices only when
`public_search_url` is present.

## Error Handling

URL generation errors do not change provider status, top-level search status,
or warning/error lists. A missing link is an output enhancement fallback, not a
search failure.

The validator must not include rejected raw URLs in public errors or logs.
Provider code should not print diagnostics to stdout.

## Testing

Add offline unit tests:

- Contract tests:
  - `FlightOfferV1` accepts `public_search_url=None`.
  - `FlightOfferV1` accepts validated provider URLs.
  - `FlightOfferV1` rejects unsafe URL strings.
  - schema export includes `public_search_url`.

- Provider URL builder tests:
  - Traveloka one-way and round-trip reuse existing builder.
  - Google public search URL is deterministic and contains route/date.
  - Skyscanner one-way and round-trip route URLs use `yymmdd` dates.
  - passenger count behavior is deterministic.

- URL validator tests:
  - accept each provider's generated URL.
  - reject `http`.
  - reject hostile hosts and protocol-relative URLs.
  - reject userinfo and non-default ports.
  - reject internal/API paths.
  - reject sensitive query keys and values.

- Search/output tests:
  - returned offers get `public_search_url` by provider.
  - unsupported/unknown provider gets `None`.
  - ranking/deduplication behavior is unchanged.
  - MCP returns Contract V1 with the new field and keeps stdout
    protocol-clean.

- Storage tests:
  - `response_json` stores validated public search URLs.
  - invalid offer URLs are nulled before persistence.
  - `offer_observations` schema remains URL-free.
  - warning/error detail sanitizer still redacts arbitrary URL-like sensitive
    strings.
  - no `transport_deeplink`, token, cookie, header, body, challenge, or session
    marker appears in stored JSON.

Default tests must not make live provider calls.

## Acceptance Criteria

- Every returned `FlightOfferV1` has `public_search_url` set to a validated URL
  or `null`.
- Traveloka, Google Fli, and Skyscanner each have a deterministic public search
  URL strategy.
- Skyscanner `/transport_deeplink/...` remains absent from Contract V1, CLI
  output, MCP output, and SQLite storage.
- No provider raw URL, cookie, header, request body, token, challenge URL,
  session ID, raw provider payload, or browser session data is exposed or
  persisted.
- The implementation is additive and tested.
- MCP stdout remains protocol-clean.
